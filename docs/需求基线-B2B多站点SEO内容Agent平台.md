# B2B 多站点 SEO 内容 Agent 平台：需求基线

> 版本：v2.0  
> 更新：2026-07-20  
> 状态：当前唯一产品需求基线。后续开发、接口设计、数据库迁移和验收均以本文为准。  
> 适用对象：面向欧美市场的 B2B 出海独立站运营团队。  
> 文档原则：旧功能和旧数据保留，但不再以旧的“通用标题 → 泛化文章”逻辑扩展。

## 1. 产品定位

平台为多个 B2B 独立站提供隔离的 SEO 内容生产闭环：

```text
站点行业资料 / 术语 / 案例 / GSC 锚文本
                    +
关键词机会 → SERP 与竞品来源 → 信息增量分析 → 人工确认大纲
                    ↓
             H2 独立深写 → 编辑组装 → 自然内链 / 图片 → 内容资产
```

目标不是批量生成看似很长的 AI 文章，而是为每个站点建立可追溯的行业资料、竞争页面证据和内容资产。系统只承诺“符合发布条件的内容生产能力”，不承诺 Google 收录、排名或流量结果。

## 2. 当前系统与保留策略

### 2.1 现有能力

- Google 下拉词递归扩展、关键词审核、分类、词库、CSV 导出、标题库和内容版本库继续保留。
- React + Vite + TypeScript、Python 本地服务和 SQLite 为当前原型技术栈；已有项目数据不得删除。
- 旧版内容生成页面、模型配置、H2 分段生成和历史正文均保留，作为历史兼容能力，不作为新版内容生产的默认入口。

### 2.2 暂缓但不删除的能力

- Google Ads API / 真实 SV、KD 数据；
- GSC OAuth、GSC 数据同步、锚文本库；
- SERP 商业数据服务、竞品全文抓取；
- 图片服务、CMS / WordPress 发布；
- 多租户登录、成员权限、计费；
- 自动化定时任务、发布后的排名与效果归因。

这些模块在本文件中定义数据契约与接入方向，后续按阶段启用，不以假数据伪装成已接入能力。

## 3. 核心数据边界

### 3.1 隔离层级

`组织（后续） → 站点 Website → 关键词 / 来源 / 任务 / 内容资产`。

一期先实现“多站点隔离”，所有读取与写入必须带 `site_id`。多租户登录和角色权限可在后续将站点归属到组织；不得让不同站点的术语、案例、GSC 锚文本或竞品资料混用。

### 3.2 站点配置（Website Workspace）

每个站点至少维护：

| 字段 | 说明 |
| --- | --- |
| 域名、目标国家/语言 | 例如 `example.com`、`US / en-US` |
| 行业与产品范围 | 例如 LED 户外照明、国际货代、铝制幕墙 |
| 目标受众与采购场景 | 采购经理、经销商、工程承包商等 |
| 品牌语调与禁止表述 | 防止所有站点写成同一种营销腔 |
| 术语与参数字典 | MOQ、IP65、EXW、尺寸/公差/认证等；每项可标记来源 |
| 专属事实与人工案例 | 工厂背景、服务范围、可公开案例；必须由用户提供或有来源 |
| GSC 授权与锚文本库 | 后续接入；密钥加密存储，不返回前端 |

### 3.3 内容生产核心对象

- `seo_tasks`：站点、目标关键词、选定标题、当前状态、失败原因、运行日志、提示词版本。
- `source_packs`：SERP、官网、用户上传资料、研究笔记等来源；必须保存 URL/发布者/日期（如有）、抓取时间、可用性与摘要。
- `competitor_snapshots`：关键词 SERP 前 10 结果、页面结构、H2/H3、实体、内容缺口；仅保存合规抓取的必要内容和来源引用。
- `outlines` / `outline_sections`：人工可编辑的大纲。每个 H2 必须保存读者问题、写作目的、要点、资料来源、格式、需要覆盖的标题承诺和内部链接建议。
- `content_sections`：每个 H2 独立生成的正文、事实主张与来源映射、待验证项。
- `content_assets` / `content_versions`：组装后的 Markdown、HTML、图片元数据、自然内链、版本历史和导出状态。
- `anchor_text_vault`：`site_id + anchor_keyword + target_url + 数据窗口 + 指标 + 状态`；只作为写作参考，不能硬插入。

## 4. 新版内容任务工作流

### 阶段 1：创建任务与资料准备

用户选择站点、关键词和标题；系统加载行业上下文、术语字典、人工案例、已有锚文本和已批准的内部页面。

若标题包含 `Best`、`Compare`、`Pricing`、`Free`、`Features`、认证、性能、尺寸或推荐等事实承诺，系统必须先检查是否存在可用来源。资料不足时任务进入“待补来源”，不得生成泛化的选购文章冒充比较文章。

### 阶段 2：SERP 与竞品研究

通过配置的 DataForSEO、SerpApi 或其他合规 SERP 数据服务获得前 10 条自然结果；需要抓取正文时使用合规抓取器并遵守来源网站条款。分析输出：

- 搜索意图、采购阶段、受众问题；
- 竞品标题、H2/H3、内容结构、实体、参数与证据来源；
- 缺失但值得补充的信息增量；
- 不应复写或无来源支撑的主张；
- 资料覆盖不足时的补充清单。

没有 SERP 服务时，允许用户粘贴 URL、文章、产品资料或研究笔记创建来源包；页面必须明确显示“未进行 SERP 研究”，不能伪称已分析前 10 名。

### 阶段 3：大纲生成与人工确认

系统基于站点上下文、来源包和内容缺口生成 5–8 个 H2 的大纲。前端支持新增、删除、排序、编辑 H2 和每节资料绑定。用户确认前，任务不得进入正文生成。

大纲规则：

1. 每个 H2 只解决一个采购或技术决策问题。
2. `Best / Compare / Pricing / Features` 等标题承诺必须有明确 H2 和相应证据。
3. 对比、参数、价格、资质、流程、风险或资格判断使用表格/清单；事实表格必须有 `Source` 列。
4. 不按竞品顺序改写；目标是补足信息缺口并建立独立论点。

### 阶段 4：按 H2 独立深写

**一个 H2 对应一次独立写作任务**。写作输入仅包含该 H2 的章节 Brief、相关来源片段、站点字典、允许的案例和已批准内部链接建议；不把未验证的整篇竞品正文当作事实来源。

每个章节需要：直接回答该节问题、给出 B2B 使用场景/选择条件/技术参数或边界、标记无证据事实为 `[VERIFY]`、返回主张到来源 ID 的映射。不同 H2 不能重复介绍同一概念。

全文默认采用“长文模式”，目标为英文约 2,000+ 词；**不设置单个 H2 的硬性字数**。当资料不足以支持深度时，应补资料或缩小标题承诺，而不是用废话凑字数。站点或任务可显式切换为“不限字数”。

### 阶段 5：编辑组装、内链与图片

组装器只做以下事情：保留每个 H2 已通过的事实边界、去重、补过渡、生成一个 H1 和语义正确的 HTML / Markdown。不得在组装阶段新增来源外的事实。

- 内链：从当前站点的 `anchor_text_vault` 读取建议。只有在语义自然且能帮助读者时才插入 `<a href>`；无法自然提及时跳过并记录原因。
- 图片：在确认的章节节点插入 Unsplash 或图像生成服务的图片；保存来源、版权/许可信息、alt 文本和插入位置。未配置图片服务时不生成伪图片。
- 编辑检查：检查来源映射、`[VERIFY]`、标题承诺、H1/H2 结构、内部链接自然度和 HTML 可渲染性。该检查是编辑步骤，不引入面向用户的质量分数或“QA 阶段”页面。

### 阶段 6：版本、导出与后续发布

将 Markdown、HTML、来源清单、未验证项、内链和图片元数据保存为新版本。用户可在阅读页查看、复制或导出；CMS 发布属于后续独立模块。

## 5. GSC 锚文本闭环（后续模块）

1. 用户为站点完成 Google OAuth 授权。
2. 后台定时同步（默认每周一次）近 30 天 GSC 查询和页面表现，避免按每篇文章实时调用而触发配额限制。
3. 根据高曝光、低 CTR、平均排名 11–30 的规则生成候选锚文本与目标 URL；允许人工确认、禁用和删除。
4. 写作阶段只读取本地锚文本库。内链不采用绝对正则替换或强制插入。

## 6. 技术架构与迁移原则

目标架构为：Next.js（React）+ Tailwind CSS、FastAPI、PostgreSQL、Celery + Redis、WebSocket / 轮询、可插拔 LLM、合规 SERP/抓取适配器。

当前项目不进行一次性推倒重写。迁移顺序：

1. 在现有系统引入 Website、来源包、结构化大纲和任务状态；
2. 把长任务从同步 HTTP 迁到异步任务接口与前端进度；
3. 数据库从 SQLite 迁移到 PostgreSQL，并增加组织/权限；
4. 接入 SERP、GSC、图片和 CMS 适配器；
5. 评估是否引入 CrewAI。多角色应是可观测的工作流，不是为了使用框架而堆砌 Agent。

所有 API Key、OAuth Token、浏览器 Profile、数据库文件和抓取缓存不得进入 Git。GSC 令牌应加密存储，日志不得输出密钥或完整授权响应。

## 7. 分期开发路线与验收

### P0：需求与数据基础

- 多站点 Workspace、站点资料/术语/案例管理；
- 关键词、标题、旧内容资产按站点隔离；
- 内容任务、来源包和大纲数据模型；
- 旧功能保持可用。

**验收：** 切换站点后不会读取其他站点资料；可创建带来源的内容任务并保存完整大纲。

### P1：研究与人工大纲

- SERP 适配器接口、手工来源包和竞品结构分析；
- 资料充分性判断；
- 可拖拽、可编辑、可确认的大纲页面。

**验收：** `Best / Pricing / Compare` 且资料不足的任务不能进入正文；确认大纲后每个 H2 都有独立目的与来源绑定。

### P2：深度写作与内容资产

- H2 独立异步生成、章节主张来源映射、组装与版本；
- Markdown/HTML 阅读、自然内链建议、图片占位/适配器接口；
- 任务日志和失败恢复。

**验收：** 每个 H2 有独立运行记录；组装不新增无来源事实；同一任务失败不覆盖旧版本。

### P3：GSC、锚文本和发布集成

- GSC OAuth、定时同步、锚文本候选与人工管理；
- 自然内链插入、图片服务、CMS 导出/发布适配器。

**验收：** GSC 只按计划同步；无法自然融入的锚文本不会被强插；导出版本含完整 HTML、图片和来源信息。

## 8. 明确禁止项

- 无来源时生成虚假的产品参数、认证、价格、案例、排名、测试结果或客户成果；
- 用通用模板伪装为行业深度文章；
- 为凑字数让多个 H2 重复同一观点；
- 声称已抓取 SERP、已接入 GSC 或已获得真实 SV/KD，实际却没有对应数据服务；
- 将不同站点的资料、案例、锚文本或授权数据混用；
- 将 Cookie、Token、浏览器 Profile、数据库或密钥提交至 Git。
