# SEO 中控系统三期：内容生成系统开发文档

> 版本：v1.0
> 更新：2026-07-17
> 范围：将已选定的 SEO 标题转化为可审核、可版本管理、可交接发布的内容草稿。
> 前置依赖：一期关键词库与审核、二期标题生成与标题库、AI 配置中心。
> 目标市场：默认外贸 Google SEO 的美国英语市场（`en-US`）；其他国家必须按指定语言区域本地化。

## 1. 模块定位

内容系统是“内容生产工作台”，不是一键批量发文器。它把关键词、搜索意图、已选标题、Google 搜索参考、品牌规则和人工要求，沉淀为可追溯的内容资产与版本。

```text
已审核关键词
      ↓
已选定标题（一期/二期）
      ↓
内容 Brief → 大纲 → 分段正文 → SEO/事实/品牌审核
      ↓
人工编辑确认 → 待发布版本
      ↓
交给 CMS 发布模块（后续）
```

本期只生成和管理内容草稿；不直接向 WordPress、Shopify 或任何 CMS 发布，也不承诺排名、收录或流量结果。

## 2. 目标与边界

### 2.1 核心目标

1. 从一个已选标题创建一条内容资产，保留其关键词、标题、目标市场和所有生成来源。
2. 先生成可人工编辑的内容 Brief 与大纲，再按章节生成正文，避免一次性不可控地生成长文。
3. 每次 AI 生成都成为独立版本，可对比、恢复、审核和导出。
4. 对草稿进行可解释的 SEO 完整性检查：搜索意图、主关键词覆盖、标题层级、Meta、FAQ、内链建议、重复风险与事实风险。
5. 为后续 CMS 发布、GSC 表现分析、内容优化、内链锚文本提供稳定的数据对象。

### 2.2 明确不做

- 不绕过 Google 验证码、登录保护或访问限制抓取页面。
- 不把 AI 生成内容标记为“事实正确”“原创无重复”或“必然排名”。
- 不自动复制、轻改或拼接竞品文章；竞品资料仅用于理解搜索意图、信息缺口和结构。
- 不在未通过人工审核时自动发布、更新或删除线上文章。
- 不把未经核验的价格、法规、医学、金融、安全、统计数据写成事实。
- 不用单一篇文章覆盖多个彼此冲突的搜索意图；应拆分为不同内容资产。

## 3. 核心对象与关系

| 对象 | 说明 | 关键关系 |
| --- | --- | --- |
| 内容资产 `content_asset` | 一篇计划中的文章/落地页的长期容器 | 关联一个项目、站点、市场与当前选定标题 |
| 内容 Brief `content_brief` | 生成正文前的写作说明和 SEO 策略 | 每个内容资产可有多个版本，只有一个当前版本 |
| 大纲 `content_outline` | H1/H2/H3、章节目标与字数预算 | 归属一个 Brief 版本，可被人工调整 |
| 内容版本 `content_version` | 可编辑的完整正文快照 | 一个内容资产有多个版本，只有一个当前工作版本 |
| 段落生成记录 `content_generation_run` | 一次 AI 请求、模型、提示词和输出摘要 | 关联 Brief/大纲/内容版本，记录来源与失败原因 |
| 审核结果 `content_review` | SEO、事实、品牌、合规、人工审核 | 不能覆盖原文，必须记录规则/提示词版本 |
| 来源资料 `content_source` | 用户输入、URL 摘要、SERP 标题快照、引用来源 | 仅保存允许使用的资料与获取时间 |

### 3.1 一对一和一对多约束

1. 一个**已选标题**在同一项目下最多对应一个未归档内容资产；若要重写，应创建内容版本，不重复建资产。
2. 一个内容资产可有多个 Brief、大纲与内容版本，但同一时刻只能有一个 `current` Brief、一个 `current` 大纲和一个 `working` 内容版本。
3. 一个关键词可关联多篇内容资产时，必须显示“可能关键词蚕食”并要求说明用途差异。
4. 内容资产删除采用软删除；正文、审核与生成记录不可因前台删除而立即物理删除。

## 4. 状态机与权限

### 4.1 内容资产状态

```text
planned → briefing → outlining → drafting → in_review → approved → ready_to_publish
                              ↓                 ↓
                           blocked            needs_revision
                                                   ↓
                                                drafting
```

| 状态 | 含义 | 可执行动作 |
| --- | --- | --- |
| `planned` | 已从选定标题创建，尚未形成 Brief | 编辑基础信息、生成 Brief |
| `briefing` | Brief 正在编辑或等待确认 | 生成/编辑 Brief、提交确认 |
| `outlining` | 正在编制大纲 | 生成/编辑/锁定大纲 |
| `drafting` | 可生成或编辑正文 | 分段生成、人工编辑、运行检查 |
| `in_review` | 已提交审核 | 审核、退回、批准 |
| `needs_revision` | 审核发现问题 | 创建新版本、修订 |
| `approved` | 内容质量已确认 | 标记待发布、导出 |
| `ready_to_publish` | 后续 CMS 模块可消费 | 发布、排期、回退为修订 |
| `blocked` | 缺资料、权限或合规原因无法继续 | 记录原因、补资料、解除阻塞 |
| `archived` | 不再使用但需保留历史 | 恢复、查看历史 |

### 4.2 角色权限（首版）

| 动作 | SEO 策略 | 内容编辑 | 审核/发布 | 只读 |
| --- | --- | --- | --- | --- |
| 建内容资产、生成 Brief/大纲 | 是 | 是 | 是 | 否 |
| 生成/编辑正文 | 是 | 是 | 可查看 | 否 |
| 修改品牌规则/AI 分配 | 项目管理员 | 否 | 否 | 否 |
| 提交/执行审核 | 可提交 | 可提交 | 是 | 否 |
| 标记 `approved` / `ready_to_publish` | 否 | 否 | 是 | 否 |

没有用户与权限系统时，首版保留 `created_by`、`updated_by`、`approved_by` 字段和权限校验接口，前台以本地管理员模式运行；不得据此省略审计字段。

## 5. 用户流程

### 5.1 从标题库进入内容工作台

1. 用户在“标题库”选择一条 `selected` 标题，点击“创建内容”。
2. 系统检查标题、主关键词、目标市场和项目是否完整；不完整时提示补全，不能建立空内容。
3. 系统创建 `content_asset`，状态为 `planned`，并复制当时的标题快照。后续改标题不应悄悄覆盖历史版本。
4. 进入内容工作台，自动带入主关键词、分类、搜索意图、标题、Google 标题参考与项目品牌规则。

### 5.2 Brief → 大纲 → 正文

1. 用户确认目标读者、商业目标、内容类型、语气、禁用词、产品/服务事实、转化动作和字数范围。
2. AI 生成 Brief；人工可逐项编辑并确认。
3. 以确认的 Brief 生成大纲；用户可以新增、删除、拖拽章节并调整标题层级。
4. 系统只在大纲被锁定后开放“生成正文”。默认按章节生成，每个章节生成后可编辑、重试或重新生成。
5. 用户可对单章节、全稿或指定问题运行审核；审核通过后人工将内容送入 `approved`。

### 5.3 版本与回退

- 每次“整篇重新生成”“使用修订意见改写”或“从已审核稿重新编辑”均创建新 `content_version`。
- 普通的连续人工输入可在当前工作版本自动保存；提交审核时冻结该版本快照。
- 可比较任意两版的标题、字数、章节、Meta、正文差异和审核结果；恢复旧版会复制为新工作版本，不直接篡改旧版本。

## 6. 页面规划

### 6.1 内容规划页 `/content`

面向选题和生产排期，字段包括：标题、主关键词、内容类型、状态、负责人、优先级、目标市场、当前版本、最后更新时间、审核状态和计划发布时间。

主要操作：从标题库创建内容、搜索/筛选、设置优先级、分配负责人、进入工作台、归档、导出清单。首版不提供批量 AI 生成按钮。

### 6.2 内容工作台 `/content/:contentId`

页面分为以下区域：

1. **上下文栏**：主关键词、已选标题、市场/语言、内容状态、目标字数、最近保存时间、当前版本和审核状态。
2. **Brief 面板**：读者、意图、内容承诺、转化目标、事实资料、禁用词、品牌语气、竞争内容洞察与必须覆盖的问题。
3. **大纲编辑器**：层级树、章节目标、预计字数、关键实体/问题、生成状态；支持人工排序和锁定。
4. **正文编辑器**：Markdown/富文本二选一（首版推荐 Markdown 作为存储格式、富文本作为编辑展示）；可按章节生成、插入图片/链接占位、查看版本差异。
5. **质量检查抽屉**：SEO 完整性、事实待核、品牌/合规、重复风险、内链机会、未完成章节；每条问题须有证据和处理建议。
6. **版本历史**：版本号、创建方式、模型/Skill、作者、字数、审核结论、创建时间、恢复/比较操作。

### 6.3 内容库 `/content-library`

内容库展示所有未归档内容资产，支持按站点、市场、状态、类型、负责人、主关键词、日期和审核结论筛选。每条记录可下钻到工作台；发布 URL、GSC 数据和排名字段在后续模块接入前显示“未连接”，不能填造数据。

## 7. Brief 规格

### 7.1 必填字段

| 字段 | 说明 |
| --- | --- |
| `primary_keyword` | 来自标题关联的已审核关键词，保存快照 |
| `selected_title` | 已选标题快照；正文 H1 默认使用它，但可人工改写并保留原因 |
| `locale` / `country` | 如 `en-US` / `US`，决定语言、拼写和示例语境 |
| `search_intent` | informational / commercial / transactional 等 |
| `content_type` | guide、comparison、service、category、landing_page、faq 等 |
| `target_audience` | 美国本地读者的角色、经验和典型场景 |
| `content_goal` | 信息教育、产品调研、询盘、订阅或其他可选目标 |
| `word_count_target` | 建议目标范围，不把字数当作质量保证 |

### 7.2 可选但强烈建议字段

- 产品/服务事实、品牌术语和必须保留的卖点。
- 禁用表述、法律/行业限制、不可承诺事项。
- 用户提供的资料、参考链接、专家访谈摘要和可引用的数据。
- Google 前 20 自然标题的快照与收集时间（仅作结构/意图参考）。
- 次要关键词、问题词、实体词和内部链接目标页面。
- CTA 类型与落地页目标；没有真实 URL 时只保存占位，不生成虚构链接。

### 7.3 Brief 生成规则

AI 只能根据已给输入提出“待确认”建议，不能假定不存在的品牌事实。所有不确定事实必须标为 `needs_source`。生成结果必须是结构化 JSON，再由服务端保存为字段和可读摘要。

## 8. 大纲与正文生成

### 8.1 大纲规则

1. 每篇内容只有一个 H1；默认取已选标题。
2. H2/H3 必须服务于搜索意图与读者问题，不以堆砌关键词为目标。
3. 对比/购买意图内容须明确比较维度和选择标准；信息型内容应优先解释、步骤、限制和 FAQ。
4. 每一节保存 `section_goal`、`estimated_words`、`must_cover`、`source_requirements` 和 `status`。
5. 大纲锁定后正文才可自动生成；解锁会提示已生成章节可能需重新审核。

### 8.2 分段生成规则

每次 AI 请求最多生成一个章节或小批连续章节，并传入：确认 Brief、当前大纲、已经完成的相关章节摘要、章节目标、允许资料、品牌规则和禁止项。

请求不得携带无关项目内容、API Key、未授权客户资料或整库关键词。输出须保存模型提供商、模型名、Skill/提示词版本、输入资料 ID、字符数、耗时和错误摘要。

### 8.3 三个 AI 的使用方式

系统复用“AI 配置”中的 ChatGPT、Gemini、DeepSeek 三个独立连接，但每一次内容生成任务只能明确指定一个主生成模型。可选的协作模式：

| 角色 | 推荐用途 | 输出要求 |
| --- | --- | --- |
| 主生成模型 | Brief、大纲、章节正文 | 输出可编辑草稿与结构化元数据 |
| 审核模型 | 事实风险、意图匹配、可读性、品牌规则 | 只返回问题与建议，不直接覆盖正文 |
| 改写模型 | 按已确认的修订意见改写指定章节 | 必须创建新版本或章节修订记录 |

首版不让三家模型同时撰写同一整篇文章，避免风格割裂和成本不可控。若后续增加“多模型对比”，必须清楚标记每段来源和人工采纳结果。

### 8.4 美国英语写作要求

- 默认用自然的 `en-US` 拼写、表达和案例语境；不要把中文营销话术直译成英文。
- 保持主关键词自然出现于 H1、引言或核心 H2，禁止机械重复。
- 不使用无法证明的 “#1”“guaranteed”“best ever”等绝对化承诺。
- 对价格、法律、合规、医疗、金融与性能结论必须要求用户来源或标记待核。
- 内容应回答搜索者问题，而不是只罗列产品卖点。

## 9. 质量审核与评分

### 9.1 审核维度

| 维度 | 自动检查 | AI/人工审核重点 |
| --- | --- | --- |
| 搜索意图 | 类型、章节覆盖、CTA 是否冲突 | 是否真正解决搜索需求 |
| SEO 结构 | H1 数量、层级、Meta、关键词覆盖、FAQ | 是否存在堆词、标题党、主题跑偏 |
| 事实与来源 | 无来源数字、绝对化词、外链占位 | 事实是否有资料支撑、是否需要法务/专家确认 |
| 品牌与合规 | 禁用词、语气、敏感类别 | 是否符合品牌与行业规则 |
| 可读性 | 段落长度、列表、句子长度提示 | 逻辑、可扫描性、美国本地表达 |
| 独特性 | 与同项目内容标题/摘要相似度 | 是否只是改写竞争对手或造成关键词蚕食 |
| 发布准备度 | 必填字段、审核状态、链接占位 | 是否允许进入 CMS 发布流程 |

评分只用于定位风险和排序，不能自动将内容批准。每项审核输出 `pass`、`warning`、`blocker`，附带证据位置、规则版本、建议和是否需要人工确认。

### 9.2 最小发布前门槛（为后续 CMS 预留）

以下任一项未通过时不得标记 `ready_to_publish`：没有已选标题、没有主关键词和市场、正文未完成、存在 `blocker`、尚未人工批准、或没有可识别的当前版本。CMS 模块还会额外验证站点、URL、图片、链接和权限。

## 10. 数据库设计

现有 `keywords`、`keyword_title_candidates` 和 `title_generation_jobs` 保持不变；内容模块新增以下表。所有外键均带 `project_id`，并默认软删除。

### 10.1 `content_assets`

| 字段 | 类型/约束 | 说明 |
| --- | --- | --- |
| `id` | INTEGER/UUID PK | 内容资产 ID |
| `project_id` | FK NOT NULL | 项目 |
| `site_id` | FK NULL | 后续站点模块接入 |
| `keyword_id` | FK NOT NULL | 主关键词 |
| `selected_title_candidate_id` | FK NOT NULL | 创建时的已选标题 |
| `title_snapshot` | TEXT NOT NULL | 创建时标题，不被后续候选改动覆盖 |
| `locale`, `country_code` | TEXT NOT NULL | 目标市场 |
| `content_type` | TEXT NOT NULL | 内容类型 |
| `status` | TEXT NOT NULL | 状态机值 |
| `priority` | INTEGER | 优先级 |
| `current_brief_id`, `current_outline_id`, `current_version_id` | FK NULL | 当前工作对象 |
| `created_by`, `updated_by`, `approved_by` | TEXT NULL | 审计归属 |
| `created_at`, `updated_at`, `archived_at`, `deleted_at` | TIMESTAMP | 时间与软删除 |

唯一约束：`(project_id, selected_title_candidate_id)` 在 `deleted_at IS NULL` 时唯一。

### 10.2 `content_briefs`

存储结构化 JSON 字段和可读摘要：读者、意图、内容目标、关键词策略、事实资料、品牌语气、禁用项、CTA、字数范围、SERP 参考快照、确认状态、版本号、生成来源与提示词版本。

### 10.3 `content_outlines` 与 `content_outline_sections`

`content_outlines` 存储资产、Brief、版本、状态（draft/locked/superseded）与模型来源；`content_outline_sections` 存储顺序、层级、标题、章节目标、预计字数、必须覆盖内容、资料要求和生成状态。章节排序使用整数序号，调整时在事务内重排。

### 10.4 `content_versions`

| 字段 | 说明 |
| --- | --- |
| `version_number` | 同一资产内递增，不复用 |
| `body_markdown` | 当前全文 Markdown 快照 |
| `body_html` | 可选渲染缓存，不能作为唯一正文来源 |
| `meta_title`, `meta_description`, `slug_proposal` | 发布前可编辑的 SEO 字段 |
| `word_count`, `reading_time` | 服务端计算 |
| `based_on_version_id` | 重写/恢复来源 |
| `created_method` | manual / ai_section / ai_full_draft / revision / restore |
| `status` | working / submitted / approved / superseded |

正文中不得保存 API Key、完整系统提示词或敏感原始授权数据。

### 10.5 `content_generation_runs`

记录每个生成/审核调用：资产、版本、章节、任务类型、provider、model、Skill/提示词版本、输入哈希、输出哈希、状态、令牌/成本（若服务返回）、开始/结束时间、脱敏错误摘要。实际提示词可保存模板 ID 与参数快照；客户机密文本仅在项目允许且加密时保存。

### 10.6 `content_reviews` 与 `content_review_issues`

审核表记录审核对象、维度、规则版本、提供商、总体结论与人工决定；问题表记录严重度、证据位置（标题/章节/文本范围）、问题说明、建议、状态（open/accepted/fixed/ignored）与处理人。

### 10.7 `content_sources`

每条资料记录类型（用户资料、SERP 标题快照、允许引用 URL、内部链接页、人工备注）、来源 URL/文本摘要、授权/可信度状态、抓取/录入时间和关联章节。禁止把未授权全文当作训练或直接改写来源。

## 11. API 规划

## 12. AI Skill 与提示词规范

内容系统应新增/复用以下 Skill，且每个 Skill 必须版本化：

| Skill | 输入 | 输出 | 禁止事项 |
| --- | --- | --- | --- |
| `content-brief` | 标题、关键词、市场、资料、目标 | 结构化 Brief | 编造品牌事实、搜索量、排名 |
| `content-outline` | 已确认 Brief | H1/H2/H3 与章节目标 JSON | 复制竞品目录、无关堆词 |
| `content-section-writer` | Brief、大纲章节、允许资料 | 单章节 Markdown 与待核声明 | 伪造引用、整篇一次性失控生成 |
| `content-seo-review` | 版本与规则 | 问题清单和证据位置 | 直接把草稿标记发布通过 |
| `content-reviser` | 审核问题与指定章节 | 新章节/新版本建议 | 静默覆盖人工原稿 |

AI 返回格式异常、超时或内容为空时，生成任务必须标记失败并保留已有版本；前端显示可重试，不得清空编辑器。

## 13. 测试计划（红绿测试）

每一项实现先在 `tests/` 写失败测试，确认红灯后再写主代码。最小验收覆盖：

1. 未选标题、未审核关键词、跨项目标题均不能创建内容资产。
2. 已选标题创建内容后，标题与关键词快照正确保存且再次创建被阻止。
3. 每个内容资产只能有一个当前工作版本；恢复旧版会产生新版本号。
4. 未锁定大纲不能调用章节生成；生成失败不丢失已保存章节。
5. 模型输出无效 JSON、超时、无配置、跨项目访问都返回明确错误和任务日志。
6. 所有生成记录显示 provider、model、Skill 版本与来源标签；API Key 永不出现在响应、日志或数据库正文。
7. 审核出现 `blocker` 时不能批准为 `ready_to_publish`。
8. 内容库筛选、状态变化、软删除、版本比较和 Markdown 导出可用。
9. React 页面具备加载、空状态、保存中、生成中、失败、无权限、版本冲突与审核阻塞状态。
10. 模块测试、全量测试与前端构建全部通过后才可提交。

## 14. 分期实施建议

### P3.1 内容资产、Brief 与大纲（建议先做）

- 标题库“创建内容”入口。
- 内容资产/Brief/大纲的数据表、API、React 页面。
- 单一 AI 主生成模型，生成 Brief 与结构化大纲。
- 人工编辑、锁定大纲、基础版本历史。

### P3.2 分段正文与版本

- 章节生成、Markdown 编辑器、自动保存、重试与版本比较。
- 模型来源标签、生成运行日志、Markdown 导出。
- 基础 SEO 检查：H 标签、字数、主关键词、Meta、FAQ。

### P3.3 审核与内容准备度

- 事实/品牌/SEO 审核问题中心。
- 人工批准流、阻塞规则、待发布交接状态。
- 内链目标与锚文本占位，为后续内链模块提供输入。

### P3.4 后续集成（不在本期实现）

- CMS 草稿创建、定时发布、回滚和实际 URL 回写。
- GSC/排名数据回流，形成“内容发布后 7/14/30 天效果复盘”。
- 内容优化建议、关键词蚕食判断、历史表现驱动的内容更新任务。

## 15. 验收标准

内容系统完成 P3.1 时，用户可以从一个已选标题创建内容资产，编辑并确认 Brief，生成和人工调整大纲，刷新后仍可看到相同资产与历史版本。所有 AI 输出来源可追溯，所有敏感配置不泄露；系统不会自动发布，也不会将草稿宣称为已排名或事实已核验。
